## 🏗️ Winston (Architect)

Concordo com Amelia sobre ambos os pontos. Deixe eu expandir:

### Sobre TP/SL Extraction

A `setPositionTpSl` manipula `PendingOrder` e `OrderHistoryItem` — domínios que pertencem ao `ordersSlice`. A lei de **SRP** diz "uma responsabilidade por módulo". Isso significa que `computeTpSlUpdate` deveria viver em `src/lib/trading/order-transitions.ts` (novo arquivo) e não em `transitions.ts` que é de posição.

Estrutura proposta:
```
src/lib/trading/
├── transitions.ts          # Position transitions (close, open, flip)
├── position-adjust.ts      # Position adjustments (add, reduce, size)
├── order-transitions.ts    # NEW: Order transitions (tp/sl, cancel)
├── validation.ts
└── position-builders.ts
```

Isso mantém cada arquivo <500 linhas (a lei) e cada um com SRP.

### Sobre Hedge Mode

Spike técnico — modelo de dados para múltiplas posições:

**Opção A: Array com alias (non-breaking)**
```ts
interface SessionSlice {
  position: Position | null;        // Alias: positions[0] || null
  positions: Position[];            // NEW: all active positions
}
```

**Opção B: Breaking change (rejected pela API Freeze)**
```ts
// NÃO FAZER — quebra 40+ componentes
- position: Position | null
+ positions: Position[]
```

**Opção C: Positions manager (mais limpo arquiteturalmente)**
```ts
// src/lib/trading/positions-manager.ts
export interface PositionsManager {
  getPrimary(): Position | null;
  getAll(): Position[];
  add(position: Position): void;
  remove(id: string): void;
}
```

Mas a Opção C viola a lei de "wrap third-party libs behind a thin interface" se considerarmos Zustand como lib. Na verdade, Zustand é o state manager do projeto, então é OK.

Minha recomendação: **Opção A** para o slice, com a `positions` array sendo a source of truth e `position` sendo um getter computado no compositor.

## 💻 Amelia (Dev)

Concordo com Winston. E mais um ponto: a lei de "early returns over nested ifs" afeta como escrevemos `checkPositions`. Hoje `checkPosition` tem 4 níveis de if aninhados (trailing stop → liquidation → SL → TP). Precisa ser reescrito como:

```ts
export function computePositionCheck(
  position: Position,
  currentPrice: number,
): PositionCheckResult {
  const trailing = checkTrailingStop(position, currentPrice);
  if (trailing.shouldClose) return trailing;

  const liq = checkLiquidation(position, currentPrice);
  if (liq.shouldClose) return liq;

  const sl = checkStopLoss(position, currentPrice);
  if (sl.shouldClose) return sl;

  const tp = checkTakeProfit(position, currentPrice);
  if (tp.shouldClose) return tp;

  return { shouldClose: false };
}
```

Cada sub-função tem ~4–6 linhas. Respeita a lei.
