## 💻 Amelia (Dev)

Acabei de entregar um pacote de correções que o usuário encontrou em produção. Resumo do que foi feito:

### Bug 1: Liquidação falhando em candles com mechas (wicks)

O `interpolatePrice` interpolava linearmente entre `open` de candles consecutivos. Se um candle tinha `open=50000, low=44000, close=49000`, e o preço de liquidação era 45000, a interpolação ia de 50000→49000 e **nunca** passava por 45000. A posição deveria ser liquidada, mas não era.

**Fix:** `processTick` agora retorna `candleLow` e `candleHigh` do candle atual. `checkPosition` recebe esses valores e verifica se o liq price está dentro do range `[effectiveLow, effectiveHigh]`.

### Bug 2: Ordens limit de redução não executavam

Mesma raiz. Uma ordem SHORT limit em 51000 nunca executava se a interpolação era 50000→49000, mesmo que o `high` do candle tenha sido 52000.

**Fix:** `checkPendingOrders` agora verifica se `candleLow <= limitPrice` (para long) ou `candleHigh >= limitPrice` (para short), além do preço interpolado.

### Bug 3: Slider max para flip não considerava PnL não-realizado

`calcSideMax` em `TradeControls.tsx` calculava `position.size + wallet * leverage`, ignorando se a posição atual estava em perda. Resultado: usuário arrastava o slider até o máximo, clicava, e recebia "Insufficient funds to flip".

**Fix:** Agora calcula `effectiveWallet = wallet + returnedMargin + closePnl` antes de definir o max.

### Bug 4: Links da home quebrados

A `LandingPage` usava `router.push()` em `<button>` ao invés de `<Link>`. Isso quebrava abrir em nova aba, clique do meio, SEO, e navegação sem JS.

**Fix:** Todos os botões de navegação convertidos para `<Link href="...">`.

## 🏗️ Winston (Architect)

Amelia, essas correções são tecnicamente sólidas, mas quero destacar uma decisão arquitetural importante que tomamos aqui.

A mudança na assinatura de `checkPosition` e `checkPendingOrders` para aceitar `candleLow?` e `candleHigh?` é **non-breaking** porque os parâmetros são opcionais. Isso preserva o API Freeze que documentamos. Mas adicionar parâmetros posicionais opcionais é uma porta que, se aberta demais, vira `checkPosition(price, low, high, volume, volatility, ...)`.

**Recomendação para o futuro:** Se precisarmos passar mais contexto do candle para o store, considerar encapsular em um objeto:

```ts
interface PriceContext {
  interpolated: number;
  candleLow: number;
  candleHigh: number;
}
```

Isso evita explosão de parâmetros posicionais.

Outro ponto: o `getCurrentCandle` que adicionamos em `binance-api.ts` duplica parcialmente a lógica de `interpolatePrice` (o loop de busca do candle ativo). Não extraímos um helper compartilhado porque ambas são funções puras de ~10 linhas, mas se crescermos isso, deveríamos extrair `findCandleIndexAtTime`.

## 📊 Mary (Analyst)

Vamos falar do elefante na sala: **como esses bugs passaram por 348 testes?**

Fiz uma análise rápida da matriz de cobertura:

| Camada | Cobertura | O que faltou |
|--------|-----------|--------------|
| Transições puras (`transitions.ts`) | ✅ Alta | Testa matemática isolada |
| Store unitário (`tradingStore.test.ts`) | ⚠️ Média | Só testa com preços exatos injetados |
| Engine (`useTimewarpEngine.test.ts`) | ❌ Baixa | 1 teste, mocks pesados, `interpolatePrice` mockado |
| Integração engine+store | ❌ ZERO | Nenhum teste simulava ticks reais com candles |
| E2E Playwright | ⚠️ Média | Pausa o engine e controla preço manualmente |

**Conclusão:** Nossos testes testavam **componentes**, não **comportamento de sistema**. O bug de liquidação só aparece quando você tem:
1. Candles reais com `low`/`high` fora do range `open→close`
2. Engine processando ticks consecutivos
3. Store verificando posição a cada tick

Nenhum teste existente juntava essas 3 camadas.

**Hipótese de processo:** A decomposição do monolito em slices + transições puras melhorou a testabilidade unitária, mas criou uma falsa sensação de segurança. "314 testes passando" não significa "o sistema funciona" — significa "as unidades funcionam isoladamente".

## 📋 John (PM)

Mary acertou no ponto sensível. A gente celebrava "baseline 314 passando" como se fosse um escudo, mas era um escudo de papel.

Vou propor uma mudança no nosso definition of done:

> **Nova regra:** Toda correção de bug em lógica de trading DEVE vir acompanhada de:
> 1. Um teste unitário que reproduz o bug com dados mockados
> 2. Um teste de integração que reproduz o bug com o engine simulando ticks
> 3. Um teste E2E que reproduz o bug via UI (quando aplicável)

Isso triplica o custo de correção, mas evita regressão. Olhando o que aconteceu:
- O usuário encontrou os bugs em produção
- Perdeu confiança nos testes
- Perdeu confiança no time ("como pode os testes deixarem passar tantos erros?")

**Custo de não ter testes de integração:** confiança do usuário + tempo de debug + context switch de refatorar testes em vez de entregar features.

**Decisão:** A refatoração completa dos testes que o usuário pediu é P0. Não é "quando sobrar tempo". É agora.

## 🏗️ Winston (Architect)

Amelia, o property-based test é arquiteturalmente elegante, mas tem um custo: se o generator produzir um caso que falha por causa de outro bug, o teste fica flaky. Sugiro começarmos com uma **tabela de cenários determinísticos**:

| Cenário | Candle (O/H/L/C) | Posição | Resultado Esperado |
|---------|-----------------|---------|-------------------|
| Long liq via low | 50000/51000/44000/49000 | LONG @ 50000, liq=45000 | Liquidated |
| Long safe despite low | 50000/51000/46000/49000 | LONG @ 50000, liq=45000 | Open |
| Short liq via high | 50000/56000/49000/51000 | SHORT @ 50000, liq=55000 | Liquidated |
| Limit fill via high | 50000/52000/49000/49000 | SHORT limit @ 51000 | Filled |
| Limit no fill | 50000/50500/49500/50000 | SHORT limit @ 51000 | Pending |

Isso é determinístico, rápido, e cobre 95% dos casos reais. Property-based é P2.

Outra coisa: a mudança no `calcSideMax` é um fix local no componente, mas o `margin.ts` ainda tem `calcSliderMax` com a lógica antiga. Precisamos sincronizar ou consolidar. Não faz sentido ter duas funções calculando o mesmo slider max com lógicas diferentes.
