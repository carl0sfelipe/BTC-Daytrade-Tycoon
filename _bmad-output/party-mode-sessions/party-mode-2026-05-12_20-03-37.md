## 📋 John (PM)

Vamos começar com a pergunta mais incômoda: **por que demoramos tanto para descobrir que nossos testes estavam cegos?**

Hoje corrigimos 4 bugs que estavam em produção. O usuário encontrou todos eles. Isso significa que nosso processo de qualidade falhou em 4 dimensões independentes:

1. **Liquidação via wicks** — bug de simulação (engine)
2. **Limit orders via wicks** — bug de execução de ordens (store + engine)
3. **Slider max errado** — bug de UX/cálculo (componente)
4. **Links quebrados** — bug de navegação (UI)

Nenhum desses bugs é trivial. E nenhum deles foi pego pelo nosso pipeline de 348 testes. Isso me leva a uma conclusão dura: **nosso pipeline testa o que é fácil de testar, não o que é importante.**

### Proposta de mudança de processo

Vou propor uma reestruturação do nosso definition of done:

| Antes | Depois |
|-------|--------|
| "Testes passando" | "Testes unitários + integração + E2E crítico passando" |
| "Build limpo" | "Build limpo + typecheck + lint + integration suite" |
| "Code review" | "Code review + test review (reviewer deve entender o que o teste prova)" |

E mais importante: **todo PR que toca em tradingStore.ts ou tick-processor.ts DEVE ser acompanhado de um teste de integração que simula pelo menos 10 ticks consecutivos com candles reais.**

## 🏗️ Winston (Architect)

Concordo com a proposta do John, mas quero adicionar uma camada arquitetural. O problema não foi só falta de testes — foi **falta de fronteiras que forcem a escrita de testes.**

Quando decompomos o monolito em slices + transições puras, ganhamos testabilidade unitária. Mas perdemos visibilidade de integração. As funções puras são tão isoladas que é possível testar cada uma sem nunca perceber que a composição está errada.

**Exemplo:** `interpolatePrice` passa em todos os seus testes. `checkPosition` passa em todos os seus testes. Mas juntos, eles falham em produção porque `interpolatePrice` não retorna wicks e `checkPosition` só olha o preço interpolado.

### Proposta arquitetural: "Integration Contracts"

Cada fronteira entre camadas deve ter um **contract test** obrigatório:

```
Engine (processTick) ──contract──▶ Store (checkPosition)
Engine (processTick) ──contract──▶ Store (checkPendingOrders)
Store (positionSlice) ──contract──▶ Transitions (computeHedgeFlip)
```

Esses contract tests não testam lógica de negócio — testam apenas que a saída da camada A é consumível pela camada B. Eles teriam pego o bug de wicks imediatamente, porque `processTick` retornava só `price` mas `checkPosition` precisava de `low`/`high`.

**Ação:** Criar `src/contracts/` com testes que verificam:
1. `TickResult` contém todos os campos que `checkPosition` e `checkPendingOrders` esperam
2. `PositionSlice` chama `computeHedgeFlip` com parâmetros válidos
3. `OrdersSlice` chama `positionSlice` actions sem deadlock circular

## 💻 Amelia (Dev)

Vou ser honesta: parte da culpa é minha. Quando refatorei o monolito em slices, eu estava tão focada em "cada função pura tem teste" que esqueci de perguntar: **"mas quem testa que o engine chama as funções certas na ordem certa?"**

A refatoração do `tradingStore.ts` foi tecnicamente correta. Mas ela introduziu uma cegueira: agora os testes unitários testam `computeClosePosition` com dados mockados perfeitos, enquanto na vida real o `currentPrice` vem de `interpolatePrice` que pode estar entre dois candles.

### O que eu aprendi hoje

1. **Nunca confie em funções puras isoladas como prova de correção.** Elas provam que a matemática está certa, não que o sistema funciona.

2. **A ordem das chamadas no `tick` importa.** Hoje descobrimos que `checkPendingOrders` deve rodar antes de `checkPosition` (ordem de execução de limit orders antes de liquidação). Se invertêssemos, uma ordem limit que deveria reduzir a posição antes da liquidação não executaria.

3. **Logs não substituem testes.** Os logs que restaurei na semana passada foram úteis para debugar, mas só depois que o usuário reportou o bug. Logs são diagnóstico, não prevenção.

### Proposta técnica: "Golden Ticks"

Criar um arquivo `src/test/golden-ticks.ts` com 50 cenários de ticks pré-computados (candles reais, posições, ordens, resultados esperados). Cada vez que alguém muda o engine ou o store, roda `golden-ticks` e compara o resultado com o snapshot. Se mudar, o dev precisa justificar.

## 🎨 Sally (UX)

Quero trazer uma perspectiva que o time técnico às vezes ignora: **os bugs de hoje não são só técnicos — são de experiência.**

O slider max errado não é só um cálculo errado. É um momento onde o usuário pensa "eu tenho dinheiro para flipar", arrasta até o máximo, clica, e recebe "Insufficient funds". A sensação é de **traição da interface**. A UI prometeu algo e não entregou.

O bug dos links quebrados é pior: o usuário tenta abrir em nova aba e não consegue. Isso é tão básico que quebra a confiança. "Se os links não funcionam, o que mais está quebrado?"

### Proposta de UX: "Fail-Safe by Design"

1. **Slider com indicador de capacidade real:** em vez de só mostrar o max, mostrar uma zona verde (dentro do effectiveWallet), amarela (acima, mas possível com lucro), vermelha (impossível). O usuário vê ANTES de clicar.

2. **Validação em tempo real do flip:** quando o slider cruza `position.size`, o botão muda de "REDUCE" para "FLIP" **e** mostra um tooltip: "Flip requires $X margin from returned funds + PnL". Se não há fundos, o botão fica desabilitado com explicação.

3. **Simulação de liquidação visual:** um modo "stress test" onde o usuário pode ver onde a liquidação aconteceria no candle atual, incluindo wicks. Isso educa o usuário E testa nossa lógica de simulação.
